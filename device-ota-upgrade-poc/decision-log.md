# OTA 手动升级 Decision Log

## 2026-04-30｜PRD 异常边界与文案口径对齐 Demo

### 结论
- PRD 中需要明确区分“升级前异常”和“升级中异常”，不能只写原则，必须写清楚触发时机、承接方式、校验优先级和对应文案。
- 所有升级前 Bottom Sheet 的标题、副标题、按钮文案，以及升级中失败结果页口径，统一与当前 demo 保持一致。

### Why
- 评审时如果只讲“异常会拦截”或“失败会承接”，开发仍然无法判断具体落点、提示优先级和文案口径，容易出现实现偏差。
- 当前已经有 demo，因此 PRD 需要反向对齐 demo，确保产品、开发、评审三方看到的是同一套规则。

### How to apply
- 后续凡是调整异常场景、提示文案或校验顺序，需同步更新 demo 与 PRD。
- 开发落地时，以 PRD 中写明的异常分类、优先级和文案口径为准。

## 2026-04-30｜PRD 内聚合交互设计说明

### 结论
- 评审与开发落地主文档统一收敛为 `prd.md`。
- 页面标题、返回按钮、弹层结构、结果页结构、按钮统一等交互落地细节，不再拆独立文档，改为 PRD 内一个专门模块统一承接。

### Why
- 当前已有可演示 demo，评审时可通过 demo 补充页面表现与感知细节。
- 对当前需求体量而言，再拆一个独立交互说明文档会增加维护成本，不如直接在 PRD 内聚合，保持评审和落地只看一份主文档。

### How to apply
- 后续评审、开发、联调与验收均以 `prd.md` 为主文档。
- 若未来交互复杂度明显提升，再考虑将交互设计说明独立拆出。

## 2026-04-30｜PRD 与交互设计说明拆分

### 结论
- 评审用 PRD 改为精简表达，重点传递需求背景、产品价值、核心链路、关键规则与异常边界。
- 页面标题、返回按钮、弹层结构、结果页结构、按钮统一等开发落地细节，统一沉淀到独立的 `interaction-design.md`。

### Why
- 当前已有可演示 demo，评审时可通过 demo 补充页面表现与感知细节，PRD 不需要继续堆叠大量页面级描述。
- 开发真正容易产生偏差的内容，反而是交互落地口径与异常处理边界，拆出独立说明更利于执行。

### How to apply
- 后续评审优先以 PRD 讲清价值、链路与边界。
- 研发设计联调、开发落地与验收时，以 `interaction-design.md` 作为页面和交互实现口径。

## 2026-04-29｜OTA 展示与链路收敛

### 结论
- 本期 OTA 手动升级在展示层采用轻量化表达，不向用户展开下载、传输、安装等技术阶段；但在升级条件、退出限制和失败承接上采用更强约束。
- 竞品仅作参考，不直接照搬；借鉴小米的轻量展示方式，借鉴 viaim 的必要风险提示和明确拦截，最终以本项目录音主链路和设备升级场景为准进行裁剪。

### 设计原则
- 入口轻：升级入口收敛在设备连接页，不作为首页主功能承接。
- 过程稳：升级前严格校验条件，升级中尽量不允许用户离开升级链路。
- 表达轻：页面统一展示“升级中”，不向用户展开技术实现细节。
- 失败可恢复：明确向用户传达升级失败后当前稳定版本仍可继续使用。

### 升级前强校验条件
开始升级前需满足以下条件，任一不满足都不允许进入升级中：
- 耳机已连接 App
- 耳机放入盒内
- 盒盖开启
- 耳机靠近手机
- 蓝牙连接正常
- 网络正常
- 耳机及充电盒电量满足要求
- 当前不在录音中
- 当前不存在进行中的升级任务

其中“耳机在盒内”“盒盖开启”“电量满足要求”属于强校验条件，本期按硬门槛处理，不作弱提示放行。

### 升级说明页保留内容
升级说明页采用轻量展示，但必须保留以下信息：
- 当前版本
- 新版本
- 更新内容
- 3 条跨 F1 / F2 都稳定成立的通用升级步骤：
  1. 耳机与 App 保持连接，耳机放入盒内并保持盒盖开启
  2. 按页面提示开始升级，并让耳机靠近手机
  3. 升级期间不要离开当前页面或退出 App
- 2 条底线规则：电量满足要求（Demo 口径不低于 20%）；若本次升级失败，当前稳定版本仍可继续使用

本期不展示预计耗时，也不在主说明页展开过细的故障恢复动作，避免因 F1 / F2 共用规则而引入容易出错的实现细节。

### 前置条件不满足的承接方式
用户点击“开始升级”后，若条件不满足，统一采用底部弹出（bottom sheet / bottom popup）承接，不再使用厚重的居中失败弹窗。底部弹出只保留当前最主要的阻断原因、一句解决建议和一个“我知道了”按钮，用户关闭后停留在说明页补足条件并重新发起升级。
优先覆盖的拦截原因包括：录音中、未放盒内、盒盖未开启、电量不足、蓝牙异常、网络异常、已有进行中的升级任务。

### 页面视觉收口原则
本期 OTA Demo 的页面表达按“正式产品稿”标准收口，不改已确认链路逻辑，只做展示层减法：
- 设备页：固件版本右侧状态文案统一收口为「有新的固件」，做成更轻的一行式信息入口。
- 升级说明页：去掉厚重卡片堆叠，只保留版本、更新内容、少量必要提醒和一个主操作。
- 升级中页：只保留进度和最必要提醒，不堆重复解释文案。
- 成功 / 失败页：统一做成极简结果页，只保留主结论和按钮。
- 交互分层：前置条件不满足用底部弹出，升级中尝试退出用居中确认弹窗，以区分轻阻断提示和危险确认动作。

### 升级中页面规则
升级中页面统一展示升级进度和关键注意事项，不拆分下载、传输、安装等技术阶段。
持续提示以下内容：
- 请勿关闭盒盖或取出耳机
- 请将耳机靠近手机
- 请勿退出 App 或离开当前页面
- 升级期间请勿使用耳机相关能力
- 若升级失败，当前稳定版本仍可继续使用

### 升级中退出规则
升级中默认不允许轻易退出页面。用户尝试返回或离开时，需强提醒“离开将导致升级失败”；若用户仍确认离开，则该次升级直接按失败处理。
这样处理是为了避免把录音、拨号及其他耳机占用场景做成复杂并发规则。

### 失败结果页与触发方式
本期保留统一的升级失败结果页，承接两类场景：
1. 当次升级过程中自然失败：从升级中页直接进入失败结果页。
2. 用户强制中断 App 后再回来：不在首页或设备页直接打断，而是在用户再次进入升级链路时展示失败结果页。

失败结果页展示口径统一为：升级失败，请稍后重试；当前稳定版本仍可继续使用。

### 升级成功结果页
本期统一展示升级成功页。用户点击“我知道了”后返回设备连接页，刷新为最新版本状态。

### 电量与距离口径
- Demo 口径：耳机及充电盒电量不低于 20%。
- PRD 口径：MVP 建议按不低于 20% 设计。
- 距离提示口径：请将耳机靠近手机，避免蓝牙中断。

### Demo 最小覆盖场景
- 有新版本，正常升级成功
- 无新版本
- 前置条件不满足：录音中
- 前置条件不满足：电量不足 / 未放盒内
- 升级过程中自然失败
- 升级中强制中断，后续重新进入升级链路时展示失败结果
